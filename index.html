// The APEX Brain
Your .claude/ folder — what lives where
Claude reads the map, not the territory. Every session begins with these files — not by scanning src/. Contractors onboard from this folder alone.
your-project/
project root
├─ CLAUDE.md
Entry point. Golden rules. Points to .claude/ files. auto-read by Claude Code
└─ .claude/
Brain directory — git-tracked, never built/deployed.
├─ CONTEXT.md
Stack, services, env vars, integrations. The "what and why". session start
├─ MAP.md
Every module described. Replaces file scanning. Biggest token saver. session start
├─ SCHEMA.md
All tables, fields, types, relationships, indexes, constraints. check before any DB change
├─ TASKS.md
Numbered sprint queue with [LEAD]/[CONTRACTOR] tags. update each session
├─ PROGRESS.md
Milestone timeline + session log. Claude appends at session end. session end
├─ DECISIONS.md
Architecture Decision Records. Never re-debate settled choices. when decided
├─ CONTRACTS.md
API endpoints, shared interfaces, data contracts. on schema change
└─ commands/
Slash commands for Claude Code. Becomes /apex-start, /apex-end, etc. native Claude Code feature
├─ apex-start.md
→ runs /apex-start in Claude Code
├─ apex-end.md
→ runs /apex-end
├─ apex-schema.md
→ runs /apex-schema — validates DB change against SCHEMA.md
└─ apex-plan.md
→ runs /apex-plan — feature planning without writing code
// Standard Session Loop
How every session flows
01
Load Brain
Type /apex-start. Reads all context files silently.
02
Read Tasks
Knows what to work on. No asking. Picks next [LEAD] task.
03
DB Check
Any DB work? Type /apex-schema first. Validates against SCHEMA.md.
04
Execute
Works on task. Only reads files relevant to the task from MAP.md.
05
End Session
Type /apex-end. Updates PROGRESS, TASKS, SCHEMA if needed.

# APEX AUDIT — Existing Project Discovery # Run this BEFORE generating any .claude/ files You are auditing an existing project to build its APEX context system. Do NOT create any files yet. Just analyze and report. Run these discovery commands and report findings: 1. `find . -type f -name "*.ts" -o -name "*.tsx" -o -name "*.svelte" -o -name "*.js" | grep -v node_modules | grep -v .git | grep -v dist | head -80` 2. `cat package.json` 3. `ls -la` (project root) 4. Step 2 — Generate context (after confirming the audit)
# APEX GENERATE — After audit confirmation The audit report is confirmed. Now generate the full APEX brain. Additional context from me: [Answer the questions Claude asked in the audit] Current project status: [e.g. "In active development, about 60% done" / "Stable, just maintenance"] Project mode to set: [NEW / ACTIVE / STABLE] Team working on this: [e.g. "Just me" / "Me + 1 contractor handling frontend"] Top 5 things I need to work on next: [List them — Claude will turn these into TASKS.md entries] Now create all .claude/ files based on the audit + this context. For SCHEMA.md, read the actual schema files found (prisma/drizzle/SQL migrations). Extract real tables, fields, types, and relationships — not summaries. Files to create: - CLAUDE.md (set mode to [ACTIVE/STABLE]) - .claude/CONTEXT.md - .claude/MAP.md - .claude/SCHEMA.md ← built from real schema files - .claude/TASKS.md - .claude/PROGRESS.md (mark pre-APEX work as "prior to APEX setup") - .claude/DECISIONS.md (ask me for key past decisions) - .claude/CONTRACTS.md - .claude/commands/apex-start.md - .claude/commands/apex-end.md - .claude/commands/apex-schema.md - .claude/commands/apex-plan.md
Step 3 — Schema extraction (if DB-heavy project)
# APEX SCHEMA EXTRACT — For existing DB-heavy projects # Run this if the project has complex DB schema to fully capture Read all schema-related files: - `prisma/schema.prisma` (if Prisma) - `src/db/schema.ts` or `src/lib/server/schema.ts` (if Drizzle) - `supabase/migrations/*.sql` (if Supabase migrations) - Any `*schema*`, `*migration*`, or `*model*` files in the project Build .claude/SCHEMA.md with: 1. Every table (name, purpose in one line) 2. Every field (name, type, nullable, default, description) 3. All foreign key relationships drawn as: TableA.field → TableB.field 4. All indexes and unique constraints 5. Enum values if any 6. RLS policies summary (if Supabase) 7. A "Change Log" section (empty, to be filled as schema evolves) Flag any inconsistencies found between migrations and current schema.

// Database Schema File
SCHEMA.md — the DB source of truth
Why SCHEMA.md instead of just reading the schema file?
Raw schema files (Prisma, Drizzle, SQL) are machine-optimized. SCHEMA.md is human+AI-optimized: it adds intent, relationships in plain English, business rules, and a change log. Claude reads SCHEMA.md at the start of any DB task instead of parsing raw migration files — faster, less tokens, more context.
01
🗄 /apex-schema — Before any DB change
Run before writing any migration, query, or model change
DB
+
Schema check prompt (the /apex-schema command)
# APEX SCHEMA CHECK — Run before any DB work Read .claude/SCHEMA.md. DB change I want to make: [describe the change: new table, new column, index, relation, rename...] Before writing any migration or query, analyze: 1. Affected tables: Which tables in SCHEMA.md are touched? 2. Relationship impact: Does this break or require updating any FK relationships? 3. Constraint check: Does this conflict with existing unique/check constraints? 4. Downstream impact: Which CONTRACTS.md interfaces or MAP.md modules query these tables? 5. Migration safety: Is this a destructive change? (column removal, type change, rename) 6. RLS impact: If Supabase, are existing RLS policies still valid after this change? Propose the migration/change. Wait for my approval before writing migration files. After approval and execution, update SCHEMA.md and add a Change Log entry.
Schema drift check — when things get out of sync
# APEX SCHEMA DRIFT — Check for inconsistencies Compare .claude/SCHEMA.md against the actual schema source: - Drizzle: read `src/db/schema.ts` or `src/lib/server/schema.ts` - Prisma: read `prisma/schema.prisma` - Supabase: check `supabase/migrations/` for the latest state Report: 1. Tables in SCHEMA.md but not in actual schema (stale) 2. Tables in actual schema but not in SCHEMA.md (missing) 3. Field mismatches (type changes, added/removed columns) 4. Relationship discrepancies Update SCHEMA.md to match the actual schema. Do not change the actual schema — only update the documentation.

// Claude Code Native Feature
Slash commands — zero-friction enforcement
How Claude Code slash commands work
Create markdown files in .claude/commands/ and they become /command-name shortcuts in Claude Code. The file content is injected as the prompt automatically. No plugins. No config. Just files in a folder — git-tracked and shared with the whole team instantly.

// Using $ARGUMENTS
Pass context directly to commands
Some commands use $ARGUMENTS — a Claude Code variable that captures whatever you type after the command name. So you can type:

/apex-schema add a "processed_at" timestamp column to the cases table

...and Claude receives that description inline in the prompt. You get full context without copy-pasting anything.
// Enforcement Strategy
How to make APEX stick — without effort
Ranked by effectiveness and setup cost. The answer is: use all three layers together.
Method Effort How it works Best for
CLAUDE.md
auto-loaded		 zero Claude Code reads CLAUDE.md automatically at every session. Put your golden rules and "read .claude/ first" instruction here. It's the only guaranteed enforcement you have. Foundation. Always.
Slash Commands
.claude/commands/		 zero Create once, use forever. The team types /apex-start instead of remembering a long prompt. Git-tracked so contractors automatically have the same commands when they clone the repo. Daily workflow. Best ROI.
CLAUDE.md DB Rule
explicit gate		 zero Add one line to CLAUDE.md: "Never write a migration or query without first reading .claude/SCHEMA.md and running /apex-schema." Claude will follow this as a rule on every session. DB change protection.
Git pre-commit hook
.husky / lefthook		 low A hook that checks if any migration files were added without a corresponding SCHEMA.md update. Blocks commits if .claude/SCHEMA.md wasn't modified alongside a migration. Simple bash script. Team enforcement.
PR checklist
GitHub template		 low Add a .github/pull_request_template.md with checkboxes: "[ ] Updated .claude/SCHEMA.md if DB changed", "[ ] Updated .claude/MAP.md if new files added", "[ ] /apex-end was run". Human accountability layer. Contractor oversight.
Claude Code hooks
settings.json		 low Claude Code supports pre/post tool hooks. You can add a PostToolUse hook that fires after file edits and reminds Claude to check MAP.md. More advanced but powerful for large teams. Advanced / big teams.
// Recommended Setup (3-layer stack)
1️⃣
Layer 1 — CLAUDE.md rules (10 min setup)
Add explicit rules: "Read .claude/ before any file", "Run /apex-schema before DB work", "Run /apex-end before closing." Claude follows instructions in CLAUDE.md as hard rules. Free enforcement.
2️⃣
Layer 2 — Slash commands (15 min setup)
Create the 4 command files in .claude/commands/. Commit to git. Team clones, commands available immediately. No config, no installs. /apex-start becomes muscle memory in one week.
3️⃣
Layer 3 — PR checklist (5 min setup)
Add .github/pull_request_template.md with APEX checkboxes. Contractors must check boxes before merge. Creates accountability without needing to review every change manually.
💡
The only rule that matters in practice
In CLAUDE.md, write this exactly: "If you did not read .claude/SCHEMA.md, do not write any SQL, migration, or ORM query. No exceptions." Claude treats this as a hard gate and will ask you to run /apex-schema if it hasn't loaded the schema yet.
