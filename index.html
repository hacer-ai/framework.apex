Claude Code Framework

Read the map.
Don't explore the territory.

APEX keeps a pre-digested .claude/ brain per project. Every session starts by reading context files — not scanning source code. Skills auto-invoke the right workflow. Works with Claude Code and OpenAI Codex from the same files.

✦
v3 — What's new
Skills replace slash commands (merged in Claude Code v2.1.3). Skills auto-invoke — Claude loads them only when relevant, loading metadata first and full instructions on demand. Token cost: near zero until needed. Cross-model: both Claude Code and Codex follow the open Agent Skills standard, so your .agents/skills/ folder works in both tools.

Project Setup

Bootstrap APEX on any project

Two paths. New projects get generated from description. Existing projects get audited first — Claude reads the actual codebase before generating any documentation.

⚡
Step 0 — Install the scaffold (run once per project)
Run the installer from this repo into your project directory. It creates all scaffold files in seconds. Project-owned files (CLAUDE.md, AGENTS.md, and all .agents/*.md brain files) are created once and never overwritten — even with --force. Only framework files (commands, skills) are refreshed by --force.
# Install into a target project ./scripts/install-apex.sh /path/to/project --project-name "My Project" # Install with Linear MCP setup instructions (optional) ./scripts/install-apex.sh . --linear # Refresh framework files (commands + skills) without touching project-owned brain files ./scripts/install-apex.sh . --force
✦
Step 0b — Initialize the brain with /apex-init
After installing, open Claude Code inside your project and run /apex-init. Claude reads the real codebase, fills .agents/ automatically, and asks about existing tasks before writing anything. Or use the manual bootstrap prompts below for full control.

Skills — How They Work

One format. Two directories.
Same workflow.

Every APEX workflow is a SKILL.md file. Claude Code loads them from .claude/skills/ — each is invokable as /name and, unless locked to manual-only, fires automatically when your words match its description. Codex loads the identical content from .agents/skills/. The brain files in .agents/ are shared by both.

💡
Manual vs auto-invoked — controlled by one frontmatter field
Add disable-model-invocation: true to a skill's frontmatter and only you can invoke it — Claude will never trigger it automatically. Leave it out and Claude matches it against your words. /apex-init and /apex-linear-bootstrap are manual-only because they have one-time side effects. Everything else auto-invokes on the right phrase.

Token Efficiency

Why this structure is efficient

📉
Skills load on demand
Skill bodies are NOT loaded into context until the skill fires — manually or automatically. Only the skill description enters context at startup. Zero token cost for 6 skills sitting in .claude/skills/.
🎯
apex-schema = automatic DB guard
apex-schema fires whether you remember to invoke it or not. Say "add a column" and the guard loads, reads SCHEMA.md, and blocks unsafe changes — no human reminder required.
🗺
MAP.md = no file scanning
Without MAP.md, Claude reads 5–20 source files to understand architecture per session. With MAP.md, it reads one. On a 40-file project: 80–90% fewer tokens for context loading.
🔄
PROGRESS.md = no re-explaining
PROGRESS.md stores what happened last session. /apex-start reads it and Claude resumes exactly where you left off — no recap, no "what were we working on."
Database Schema

SCHEMA.md — the DB source of truth

Raw schema files (Prisma, Drizzle, SQL) are machine-optimized. SCHEMA.md is human + AI-optimized: it adds intent, relationships in plain English, business rules, and a change log. The apex-schema skill reads this before any DB work.

⬡
The CLAUDE.md DB rule — the simplest enforcement
Add this exact line to CLAUDE.md: "Never write a migration, query, or ORM model without first reading .agents/SCHEMA.md." Claude Code treats CLAUDE.md as hard rules. Combined with apex-schema auto-invocation, you get two layers of DB protection for zero ongoing effort.

Practical Usage

When to use which tool

✦
Use Claude Code when…
Interactive sessions, complex multi-step tasks, architecture decisions, code review, contractor onboarding. Claude Code's skill auto-invocation and CLAUDE.md rules make it the better choice for open-ended work where you need judgment.
⧉
Use Codex when…
Batch tasks, parallel feature branches, well-scoped automated work. Codex's cloud execution and multi-agent parallelism makes it strong for "run 5 of these tasks simultaneously while I work on something else."
💡
The handoff pattern
Plan in Claude Code (interactive, architectural) → run bulk tasks in Codex (parallel, cloud) → review and merge in Claude Code. Both read the same .agents/ brain, so context is never lost between tools. Run apex-end in Claude Code before handing off to Codex so TASKS.md is current.
Linear Integration

One source of truth.
TASKS.md + Linear in sync.

The strategy is simple: TASKS.md is the read layer (zero API cost every session), Linear is the visibility and collaboration layer. Linear IDs are stored directly inside TASKS.md — no lookup needed. The MCP only fires when a state actually changes.

💡
The token-efficient principle
Claude reads TASKS.md locally every session — no API call. Linear only gets touched when: (1) bootstrapping a new project, (2) a task moves to In Progress, (3) a task is marked Done. Three operations, not continuous polling. The Linear ID stored in TASKS.md means Claude always knows exactly which issue to update without searching.

Token Strategy

When Linear MCP is called — and when it isn't

Operation API call? When Token cost
Read task list Never Every session start — reads TASKS.md locally 0 tokens
Bootstrap project Once Once per project — creates all issues, writes IDs back ~50 tokens / issue
Move to In Progress On change apex-linear-sync auto-invokes, 1 API update call ~80 tokens
Mark task Done On change apex-linear-sync: 1 status update + 1 comment ~120 tokens
Add new task On create apex-linear-add: 1 create call, writes ID to TASKS.md ~100 tokens
Read Linear for context Never Claude reads TASKS.md instead — Linear is write-only in APEX 0 tokens
⚡
Linear is write-only in APEX — never a read source
The most common token trap with any issue tracker integration is reading from it every session to "stay in sync." In APEX, Linear is downstream — TASKS.md is the source of truth. Claude writes to Linear when things change. It never reads from Linear to figure out what to work on. The [LIN: ENG-47] tag in TASKS.md is all the reference Claude needs to push an update.
