Shared brain for AI coding teams

Fewer tokens per session.
One source of truth across the team.

APEX keeps a pre-digested .agents/ brain per project. Every session starts by reading context files — not scanning source code. Humans, Claude Code, and Codex all read and write the same brain. Linear is the team-facing layer, kept in sync automatically.

Two outcomes: lower token consumption (a 2 KB MAP.md replaces hundreds of file reads; most sessions load < 5 KB before doing real work) and synchronized teams (non-engineers see real status in Linear without touching the repo).

✦
v4 — What's new
Symphony-aligned state machine — TASKS.md uses Todo · In Progress · Human Review · Rework · Done, mirroring OpenAI's Symphony. Migration is free if you outgrow APEX.

Bidirectional /apex-sync — pulls Linear issues with the apex label or assigned to you into TASKS.md, then pushes local state changes back. Idempotent.

/apex-tidy on Haiku — pure janitorial pass (archive Done > 14 days, compress old PROGRESS, fix ID collisions). Run it cheap: claude -p --model claude-haiku-4-5-20251001 "/apex-tidy".

PR-proof gate — a task cannot enter Done without [PR: #N] or [@<short-sha>]. Symphony's proof-of-work principle, at file scale.

LINEAR_MODE setting — source · mirror · off. Pick whether Linear is the team's source of truth or a downstream mirror.
⌬
Where APEX sits in the stack
APEX is not an orchestration platform like OpenAI's Symphony — it's the layer underneath. If Symphony is "manage work, not agents," APEX is "give the agents one brain to manage from." Solo devs and small teams (2–5 people) run APEX with zero infrastructure. Bigger teams keep APEX and graduate to Symphony on top — the brain files port directly.

✅
Select a task at session start
After /apex-start loads context, Claude shows In Progress, Human Review, and Up Next (Todo). Tell Claude which one to work on — or say "show me the full list" to see Backlog. The selected task moves to In Progress in TASKS.md (and Linear, when configured).
➕
Add a new task anytime
Say "add task: [description]" at any point. The apex-linear-add skill creates it in TASKS.md with the next T-NNN number and (if LINEAR_MODE != off) creates the Linear issue, returning the ID inline.
🔐
Done requires proof
/apex-end refuses to mark a task Done without a [PR: #N] tag (pull request) or [@<short-sha>] tag (commit). Mirrors Symphony's proof-of-work principle, at file scale.
⇄
Bidirectional Linear sync
/apex-sync pulls Linear issues with the apex label or assigned to you into TASKS.md, then pushes any local state changes back. Idempotent — safe to run any time.
Project Setup

Bootstrap APEX on any project

Two paths. New projects get generated from description. Existing projects get audited first — Claude reads the actual codebase before generating any documentation.

⚡
Step 0 — Install the scaffold (run once per project)
Run the installer from this repo into your project directory. It creates all scaffold files in seconds. Project-owned files (CLAUDE.md, AGENTS.md, and all .agents/*.md brain files) are created once and never overwritten — even with --force. Only framework files (commands, skills) are refreshed by --force.
# Install into a target project ./scripts/install-apex.sh /path/to/project --project-name "My Project" # Install with Linear MCP setup instructions (optional) ./scripts/install-apex.sh . --linear # Refresh framework files (commands + skills) without touching project-owned brain files ./scripts/install-apex.sh . --force
✦
Step 0b — Initialize the brain with /apex-init
After installing, open Claude Code inside your project and run /apex-init. Claude reads the real codebase, fills .agents/ automatically, and asks about existing tasks before writing anything. Or use the manual bootstrap prompts below for full control.

Commands vs Skills — when to use which in Claude Code
Use .claude/commands/ for things you type manually: /apex-start, /apex-end, /apex-sync, /apex-tidy. Single .md file, no frontmatter needed, shows up in /help autocomplete immediately.

Use .claude/skills/ for things Claude should trigger automatically: apex-schema fires when you say "add a column" without you typing anything. Requires a directory + SKILL.md with a description frontmatter field.

+
What it does — three steps gated on LINEAR_MODE
Resolve LINEAR_MODE from .agents/CONTEXT.md (default: mirror). off → exit silently mirror → push only (skip Step A) source → bidirectional (run all steps) ## Step A — Pull from Linear (source mode only) Query Linear for issues that match ALL: - Team ID matches CONTEXT.md - Project matches CONTEXT.md (if set) - Label includes "apex" OR assignee is current user - Status: Todo, In Progress, or In Review - Issue ID NOT already in TASKS.md as [LIN: ABC-NNN] For each new issue: 1. Compute next free T-NNN (max existing + 1, fresh per insert) 2. Map Linear status → TASKS.md section: Todo → 🔵 Todo In Progress → 🔴 In Progress In Review → 🟣 Human Review 3. Append: - [ ] T-NNN [LIN: ABC-NNN] — {title} [LEAD] ## Step B — Push state changes For each TASKS.md task with [LIN: ABC-NNN]: - Compare local section vs Linear status - If different, push local → Linear with comment "Synced via APEX — {old} → {new} on YYYY-MM-DD" - Conflict rule (source mode): if Linear's updatedAt is newer than local edit, trust Linear. Update TASKS.md to match. ## Step C — Report ✓ APEX-SYNC — YYYY-MM-DD · mode=<mode> pulled: N issues pushed: M state changes conflicts: K (resolved by trusting <Linear|local>) unsynced: TASKS.md tasks without [LIN: ...] On any failure: leave TASKS.md untouched, append errors: line. Idempotent. Safe to run repeatedly.

TASKS.md — archive old Done For every Done task with DONE date > 14 days old: - Append verbatim to .agents/archive/TASKS-YYYY-MM.md (scope by month the task was Done) - Delete from TASKS.md ## 2. Section integrity - [x] items NOT in Done → move to Done (with proof tag preserved) - Items in Done not [x] → mark [x] - In Progress items without 🔒 ACTIVO and no recent commits → flag under "needs human review" ## 3. ID collisions Scan T-NNN and ADR-NNN. If repeated: - Keep first occurrence - Renumber later occurrences to next free max+1 - Report renumbers ## 4. Formatting normalize Tag order: T-NNN → [LIN: ...] → — → title → `[LEAD]` → [CONTRACTOR: ...] Reorder if needed. Don't invent missing tags. Don't change titles. ## 5. PROGRESS.md — compress old sessions If PROGRESS.md > 150 lines: - Keep Milestones verbatim - Keep last 5 sessions detailed - Older sessions → 1 row each in Archive table (≤ 100 chars summary) ## 6. Report ✓ APEX-TIDY — YYYY-MM-DD archived: N tasks → .agents/archive/ progress: M sessions compressed renumbered: list (or "none") formatting: K lines normalized needs human review: - [bullet per item] Do NOT touch MAP.md, SCHEMA.md, DECISIONS.md (except ID collisions), CONTEXT.md, CLAUDE.md, AGENTS.md. Token Savings

Why this structure is efficient

📉
Commands load on demand
.claude/commands/ files are NOT loaded into context until you type the command. Zero token cost at session start — only the command you invoke gets loaded.
🎯

Having apex-schema in both .claude/commands/ (for manual /apex-schema) AND .claude/skills/ (for auto-invocation) means the DB guard fires whether you remember to type it or not.
🗺

🔄

Linear Integration

TASKS.md is the buffer.
Linear is the team-facing layer.

Linear holds history; TASKS.md holds attention. Linear IDs are stored directly inside TASKS.md — no lookup needed. Pick one of three modes depending on how your team works.

⚙
LINEAR_MODE — three behaviors, one setting
Set LINEAR_MODE in .agents/CONTEXT.md to control how aggressively Linear is treated as the source of truth.

source — Linear is the team's source of truth. /apex-start runs /apex-sync first to pull issues with the apex label or assigned to the current user. Recommended for teams of 2+ where non-engineers also create tickets.

mirror — TASKS.md is local-first; state changes are pushed to Linear on transitions. Reads stay zero-cost. Default. Recommended for solo work that wants Linear as a durable archive.

off — Linear unused. All Linear skills no-op. Recommended for personal projects or when MCP is not configured.
⇄
Bidirectional sync — /apex-sync
Pulls Linear issues into TASKS.md (when source), then pushes any local state transitions back to Linear. Idempotent — safe to run repeatedly. Per-task incremental pushes (when /apex-end finalizes a single task) still go through apex-linear-sync; /apex-sync is the bigger reconcile and runs automatically at /apex-start when LINEAR_MODE=source.

⚡
Reads stay zero-cost in every mode
Even in source mode, Claude never reads Linear during regular task work — it reads TASKS.md. The Linear API is only touched at session boundaries (/apex-sync) or on state transitions (apex-linear-sync). The [LIN: ABC-NNN] tag in TASKS.md is enough reference for any push.
